## Sprint 338 — Fix P0: stale WASM → leaf health always 0

### Root Cause
The `public/wasm/` directory (served by Vite dev server) contained a WASM binary from March 16 that predated the health-writing code in `tree_rasterize`. The `npm run wasm` script built fresh WASM to `wasm/` but never copied to `public/wasm/`. Vite resolves `/wasm/groundwork_sim.js` from `public/` first, so the dev server always served the stale binary.

### Investigation Path
1. Health histogram showed ALL 1011 leaf voxels had water_level=0 (should be 255 for healthy trees)
2. Rust unit test confirmed health IS correctly written in native Rust — all 14,882 leaves had non-zero health
3. Comparing grid material counts (stone=134,279 in WASM vs 128,000 in Rust) revealed the grids were different
4. Three WASM copies found: `wasm/` (fresh), `public/wasm/` (stale Mar 16), `dist/wasm/` (intermediate)
5. Copying fresh WASM to `public/wasm/` → leaf voxels immediately showed water_level=255

### Fix
- Rebuilt WASM from current Rust source
- Updated `npm run wasm` script to auto-copy output to `public/wasm/` after build
- Added `getTreeEntityHealth()` to agent API (reads Tree ECS components directly for debugging)

### Verified
- Leaf samples: `[8, 255, 233, 0]` — material=Leaf, water_level=255 (health=1.0), light=233
- Health histogram: 9800 thriving, 0 dead
- Tree entity health: species=0 (Oak), health_u8=255, stage=3 (YoungTree)
- All 111 Rust tests pass
- TypeScript compiles cleanly

### Impact
- Foliage health tinting now works — stressed/dying trees will show brown/yellow leaves
- Deep playtest health metrics will report accurate data
- The stress test crowding scenario will now show visible die-offs via health coloring
